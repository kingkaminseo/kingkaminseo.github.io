## 문제 8-1
한 회사가 최신 청구서에서 Amazon EC2 비용이 증가한 것을 관찰했습니다. 청구 팀은 몇몇 EC2 인스턴스에서 원치 않는 인스턴스 유형의 수직적 스케일링(수직 확장)이 발생한 것을 확인했습니다. 솔루션스 아키텍트는 최근 2개월의 EC2 비용을 비교하는 그래프를 생성하고 인스턴스 유형 기반의 심층 분석을 수행하여 수직적 스케일링의 근본 원인을 식별해야 합니다. 가장 적은 운영 부담으로 정보를 생성하려면 어떻게 해야 합니까?



  

### 선택지
1. AWS Budgets를 사용하여 예산 보고서를 생성하고 인스턴스 유형을 기준으로 EC2 비용을 비교합니다.

2. Cost Explorer의 세부 필터링 기능을 사용하여 인스턴스 유형을 기준으로 EC2 비용을 심층 분석합니다.

3. AWS 청구 및 비용 관리 대시보드의 그래프를 사용하여 최근 2개월 동안 인스턴스 유형 기반 EC2 비용을 비교합니다.

4. AWS Cost and Usage Reports를 생성하여 Amazon S3 버킷으로 전송합니다. Amazon S3를 소스로 Amazon QuickSight를 사용해 인스턴스 유형 기반의 인터랙티브 그래프를 생성합니다.

## 문제 8-2
한 회사가 애플리케이션을 설계하고 있습니다. 애플리케이션은 Amazon API Gateway를 통해 정보를 수신하고 Amazon Aurora PostgreSQL 데이터베이스에 정보를 저장하기 위해 AWS Lambda 함수를 사용합니다. 개념 증명 단계에서 회사는 데이터베이스에 로드해야 하는 대량의 데이터를 처리하기 위해 Lambda 할당량을 크게 늘려야 했습니다. 솔루션 아키텍트는 확장성을 개선하고 구성 작업을 최소화하는 새로운 설계를 권장해야 합니다. 어떤 솔루션이 이러한 요구사항을 충족합니까?




### 선택지
1. Lambda 함수 코드를 Amazon EC2 인스턴스에서 실행되는 Apache Tomcat 코드로 리팩터링합니다. 네이티브 Java Database Connectivity(JDBC) 드라이버를 사용하여 데이터베이스에 연결합니다.

2. 플랫폼을 Aurora에서 Amazon DynamoDB로 변경합니다. DynamoDB Accelerator(DAX) 클러스터를 프로비저닝합니다. 기존의 DynamoDB API 호출이 DAX 클러스터를 가리키도록 DAX 클라이언트 SDK를 사용합니다.

3. 두 개의 Lambda 함수를 설정합니다. 하나의 함수는 정보를 수신하도록 구성하고, 다른 하나의 함수는 정보를 데이터베이스에 로드하도록 구성합니다. Amazon Simple Notification Service(Amazon SNS)를 사용하여 Lambda 함수들을 통합합니다.

4. 두 개의 Lambda 함수를 설정합니다. 하나의 함수는 정보를 수신하도록 구성하고, 다른 하나의 함수는 정보를 데이터베이스에 로드하도록 구성합니다. Amazon Simple Queue Service(Amazon SQS) 큐를 사용하여 Lambda 함수들을 통합합니다.

### 용어 및 개념 정리 

- 백프레셔(Backpressure, 배압):  데이터 처리 시스템에서 생산자(Producer)의 속도가 소비자(Consumer)의 처리 속도9보다 빠를 때, 소비자가 감당할 수 있는 만큼만 데이터를 보내도록 생산자에게 신호를 보내 데이터 흐름을 제어하는 안정화 메커니즘
