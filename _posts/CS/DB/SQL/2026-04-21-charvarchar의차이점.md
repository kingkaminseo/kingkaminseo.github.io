## CHAR, VARCHAR의 차이점

데이터베이스에서 문자열을 저장할 때 가장 기본이 되는 두 자료형 **CHAR**와 **VARCHAR**의 여러 차이점을 알아보자.

### 주요 특징 및 차이점 정리

| 구분      | CHAR (Character)                                  | VARCHAR (Variable Character)                                |
| --------- | ------------------------------------------------- | ----------------------------------------------------------- |
| 유형      | 고정 길이 문자열                                  | 가변 길이 문자열                                            |
| 저장 방식 | 설정한 길이에 못 미치면 남은 공간을 공백으로 채움 | 실제 입력된 글자 수만큼만 저장 (+ 길이 정보 1~2바이트 추가) |
| 성능      | 비교적 빠름 (데이터의 시작점을 찾기 쉬움)         | 비교적 느림 (길이가 제각각이라 별도 계산 필요)              |
| 공간 효율 | 데이터 길이가 일정하지 않으면 낭비 발생           | 데이터 길이에 맞춰 저장하므로 효율적임                      |

### CHAR(10)

- 무조건 10Byte를 확보한다.
- 만약 5 Byte의 문자열을 저장하였다면 나머지 5Byte의 빈 공간은 공백으로 채워진다.

### VARCHAR(10)

- 실제 글자수 + 길이 정보를 저장한다.
- 최대 10 Byte의 길이를 저장할 수 있다.
- 가변적이만 최대 길이를 지정하는 이유
  1. 메모리 설계
  2. 데이터 무결성
- 예를들어 5Byte길이의 문자열을 저장하려고 한다면 5글자임을 나타내는 1Byte를 합쳐 총 6Byte를 사용하여 저장한다.

## CHAR, VARCHAR 공통점

1. 문자열을 데이터 타입으로, 싱글쿼터('')로 값을 나타낸다.
2. 대소문자를 구분한다
3. 한글은 글자당 3byte 영문자는 알파벳 하나당 1byte의 크기를 가진다.

## 언제 사용하면 좋을까?

### CHAR의 경우

> 데이터의 길이가 변하지 않는 고정값일 때 유리하다.  
> ex: 주민등록번호, 전화번호, 이름, Y/N 상태 값 등  
> {: .prompt-tip }

### VARCHAR의 경우

> 데이터의 길이가 가변적이거나 예측하기 어려울 경우 유리하다.  
> ex: 이메일 주소, 게시글 내용 등  
> {: .prompt-tip }

## 지정된 길이를 넘겼을 경우 오류 메시지

### MYSQL / MariaDB

```
Error Code: 1406. Data too long for column '컬럼명' at row 1
```

### PostgreSQL

```
ERROR: value too long for type character varying(10)
```

### Oracle

```
ORA-12899: value too large for column "테이블명"."컬럼명" (actual: 11, maximum: 10)
```
