## 왜 OR 대신 UNION을 써야 할까?

SQL에서 데이터를 조회할 때 여러 조건을 만족하는 결과를 얻기위해 흔히 OR을 사용하였다.


```sql
SELECT e.name, b.bonus
FROM Employee e 
LEFT JOIN Bonus b ON e.empId = b.empId
WHERE b.bonus < 1000 OR b.bonus IS NULL;
```
이 코드는 보너스가 1,000 미만이거나 NULL인 직원의 이름과 보너스 금액을 찾는 SQL 쿼리이다.  
다음과 같이 OR 연산자를 사용하여 쿼리를 작성할 수 있다.  
이 쿼리는 간단하고 직관적이지만, 성능 면에서는 최선의 선택이 아닐 수 있다.  

## OR연산자의 문제점?
OR 연산자를 사용하면 데이터베이스는 한 번의 스캔으로 모든 조건을 확인해야 한다.  
이 과정에서 불필요한 데이터까지 대량으로 검색하게 되고, 특히 인덱스를 제대로 활용하지 못하는 경우가 많다.  
이를 <mark>Full Table Scan</mark> 이라고 한다.

> Full Table Scan  
데이터베이스에서 쿼리 수행 시 인덱스를 사용하지 않고 테이블의 처음부터 마지막 레코드까지 전체를 하나씩 읽어 조건을 검사하는 방식  
{: .prompt-tip }

## UNION 전략

```sql
-- 보너스가 1000 미만인 경우
SELECT e.name, b.bonus
FROM Employee e 
LEFT JOIN Bonus b ON e.empId = b.empId
WHERE b.bonus < 1000

UNION ALL

-- 보너스가 NULL인 경우
SELECT e.name, b.bonus
FROM Employee e 
LEFT JOIN Bonus b ON e.empId = b.empId
WHERE b.bonus IS NULL;
```

다음과 같이 OR 연산자말고 집합 연산자인 `UNION ALL`을 사용하여 코드를 작성해보았다.  
이렇게 하면 데이터베이스는 각 쿼리를 독립적으로 최적화하고 실행할 수 있다.  
UNION으로 나누면 각각의 WHERE 조건에 맞는 인덱스를 개별적으로 활용할 수 있어 성능측면에서 이점을 가져갈 수 있다.  
`bonus < 1000`인 데이터와 `bonus IS NULL`인 데이터는 서로 겹칠 수 없으므로 UNION ALL을 사용하였다.  
UNION ALL을 사용하면 불필요한 중복을 체크하지 않아 성능 측면에서 훨씬 유리하다.

> 중복이 없다는 게 확실하다면 UNION ALL을 사용해 중복 제거 단계를 건너뛰고 성능을 더 높일 수 있겠지만 대부분의 경우 정확한 결과를 위해선 UNION이 권장한다고 한다.  
{: .prompt-tip }

## 주의할 점
### 데이터 양이 적을 때 

> 조회하고자 하는 데이터의 양이 적을 경우 오히려 두 번 조회하고 합치는 비용이 더 클 수 있다.  
{: .prompt-warning }

### 중복 데이터가 많을 때

> 조회하고자 하는 데이터가 중복이 많을 경우 UNION을 써서 중복을 제거하는 비용이 OR 연산보다 클 수 있다.  
{: .prompt-warning }
